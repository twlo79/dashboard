# 安幸收支儀表板 · Oasis Living Ledger Dashboard

將物業管理的多年記帳資料,從「每月一個 Excel/Sheet 分頁」轉換成一個**可互動、可視覺化、自動同步**的線上財務儀表板,供主管即時查閱。

整套架構強調三件事:**改動最少**(同事記帳習慣不變)、**減少錯誤率**(UUID 主鍵 + 下拉選單)、**全免費**(所有服務都用免費額度)。

## 1. 使用的服務

| 層級 | 服務 | 角色 | 費用 |
|---|---|---|---|
| 輸入層 | **Google Sheet** | 同事記帳,每月一個 `YYYYMM` 分頁 | 免費 |
| 同步引擎 | **Google Apps Script** | 自動發 UUID、把 Sheet 資料 upsert 進資料庫、Web App 端點 | 免費,無 operation 上限 |
| 資料層 | **Supabase** (PostgreSQL) | 儲存交易、RLS 唯讀安全、預建彙總 View | 免費額度 |
| 圖表 | **Apache ECharts** | 環圈圖、平均線折線、排行榜 | 開源免費 |
| 部署 | **Vercel** | 託管前端(單一 HTML),提供公開網址 | 免費 (Hobby) |
| (選用) | **Make.com** | 僅做「同步完成發 LINE 通知」等輕量任務 | 免費版 |

### 為什麼這樣選
- **同步引擎用 Apps Script 而非 Make**:資料量約 3,000+ 列,Make 免費版每月僅 1,000 operations,跑一次全表同步就會爆額度。Apps Script 免費且無此限制,並能直接當「同步按鈕」的後端。
- **資料庫用 Supabase**:解耦(網站不依賴 Google 活著)、SQL 查詢力、schema 穩定適合長期內部工具。資料量對免費額度(500MB)而言十年用不完。

## 2. 資料庫邏輯

### 主表 `ledger`

一筆交易一列(tidy format)。欄位:

| 欄位 | 型別 | 說明 |
|---|---|---|
| `uid` | uuid **PRIMARY KEY** | 唯一碼,防重複的錨點 |
| `ym` | text | 年月,如 `2026-06` |
| `item` | text | 項目 |
| `txn_date` | date | 日期 |
| `buyer` | text | 購買人 |
| `txn_type` | text | 收支類型:`支出` / `收入` / `J零用金` / `K零用金` |
| `amount` | numeric | 金額(**支出為負值**) |
| `property` | text | 房源(原始名稱,保留細顆粒度) |
| `property_group` | text | 房源群組(彙總用) |
| `cert_type` | text | 憑證類型(發票/收據/匯款…) |
| `invoice_no` | text | 發票號碼 |
| `onetime_no` | text | 一次付款編號 |
| `pay_method` | text | 付款方式 |
| `category` | text | 支出類別 |
| `filler` | text | 填寫人 |
| `note` | text | 備註 |

索引:`ym`、`property_group`、`txn_type`、`txn_date`。

### 三層防錯設計(核心邏輯)

| 防線 | 機制 | 解決的問題 |
|---|---|---|
| **UID 設 PRIMARY KEY** | 資料庫底層拒絕同 uid 第二筆 | 重複寄入(物理擋掉,非事後檢查) |
| **upsert(`on_conflict=uid`)** | 以 uid 為準,有則覆蓋無則新增 | 同事改值後資料庫跟著更新 |
| **每週對帳(reconcile)** | 比對 Sheet 與 DB 的 uid 集合,差集刪除 | 同事刪除舊資料後同步 |

> 重點:防重複交給**資料庫的 UNIQUE 約束**,不交給 Apps Script 判斷。即使腳本重送,DB 也歪不了。

### 安全:RLS 唯讀
- 開啟 Row Level Security,只給 `select` policy → 前端用的 **publishable key 只能讀、不能改**。
- 寫入只走 Apps Script(使用 **secret key**,跑在 Google 伺服器端,不外洩)。
- 結論:儀表板放上公開網址也安全,任何人都無法竄改帳目。

### 彙總 View(前端只讀這些,不拉原始 3,000 列)
| View | 用途 |
|---|---|
| `v_month_group` | 每月 × 房源群組 × 收支 |
| `v_month_total` | 每月總計(畫趨勢線/平均線) |
| `v_year_group` | 年度 × 房源群組(排行榜) |
| `v_month_category` | 每月 × 支出類別(類別分析) |

### `sync_log` 表
記錄每次同步的時間與筆數,供前端顯示「上次同步」。

## 4. 同步與自動更新

### 三種觸發方式

| 觸發 | 函式 | 範圍 | 負責 |
|---|---|---|---|
| **網站按鈕 / 每晚自動** | `syncRecent` | 近 2 個月(當月+上月) | 新增、改值 |
| **每週日深夜** | `reconcile` | 全表 ≥ START_TAB | 刪除對帳 |
| **同事編輯當下** | `onEdit` | 該列 | 自動補 UUID |

### 為什麼是「近 2 個月」而非全表
- 歷史月份已結清不會變,每晚重掃浪費。
- 只掃 1 個月會漏掉**月初補登上個月**的帳,所以掃「當月+上月」最穩。
- **刪除**無法靠近月同步偵測(刪了舊月份的列掃不到),故另設**每週全表對帳**補上。

### 同步按鈕做什麼
網站右上「同步」按鈕 → 呼叫 Apps Script 的 Web App 端點(`doGet`)→ 立即跑 `syncRecent` → 回傳結果 → 網站重抓 Supabase 並重畫圖、更新「上次同步」時間。給「等不及每晚自動、想當下刷新」的主管用。

### upsert 技術細節
```
POST {SUPABASE_URL}/rest/v1/ledger?on_conflict=uid
Headers: apikey / Authorization = SECRET_KEY
         Prefer: resolution=merge-duplicates,return=minimal
Body: [ {uid, ym, item, ...}, ... ]   // 每批 500 筆
```

## 5. Google Sheet 設定

### 必要設定(從 2026-06 起實施)
1. **分頁命名**:一律 6 位數字 `YYYYMM`(如 `202606`)。腳本靠 tab 名鎖定月份,改名會找不到。
2. **欄位順序固定**(A→N):
   ```
   A 項目  B 購買日期  C 購買人  D 收入支出  E 價格  F 房源
   G 發票或證明文件  H 發票號碼  I 一次付款編號  J 付款方式
   K 支出類別  L 填寫人  M 備註  N UUID
   ```
3. **UUID 欄(N 欄)**:由 `onEdit` 自動產生,**任何人都不可手改**——它是同步、防重複、防改、防刪的鑰匙。
4. **下拉選單(資料驗證)**:房源(群組)、付款方式、支出類別、收入支出都設成下拉,**同事只能選不能打**,杜絕同源異名(例:開封整棟 / 開封-4F 名稱不一致)。

### 維護鐵律(否則同步腳本會破)
- ✅ tab 名永遠 `YYYYMM`
- ✅ 新月份一律複製固定模板,欄位順序不可動
- ✅ UUID 欄不可手改
- ❌ 不可在中間插欄、改欄名、改順序

### 舊資料處理
- 2026-05 以前的舊分頁(結構不一致:早期 10 欄 / 中期「收入or支出」/ 近期 13 欄)**已一次性清洗並匯入 Supabase**,腳本不再讀取,維持現狀。
- 清洗動作:對齊欄位、用金額正負號補回早期缺失的收支欄、統一用字、建立房源群組對照、每列發 UUID。

## 6. 筆記

### 房源分類邏輯
房源原始名稱保留(如「開封-2F / 開封-3F」是真的不同單位,不可合併),另設 `property_group` 欄做彙總(開封各樓 → 群組「開封」)。儀表板用群組彙總、明細查原始房源,兩種顆粒度並存。

### 收支類型分類邏輯(重要)
讀每筆的「收支類型」欄,分三類處理:
- **支出**:真正對外花費 → 計入費用
- **收入**:有錢進帳(如解約金、退稅) → 不計入費用
- **J零用金 / K零用金**:**錢包之間搬移**(撥款 / 結清),**不計入收支**

> ⚠️ 為什麼零用金撥補要獨立:撥款(如 +150,000)只是把錢放進零用金錢包,**還沒花掉**;實際花費已分散記在各筆「支出」(付款方式寫 `jessica零用金`)。若把撥款也算進去 = 同一筆錢算兩次(重複計算)。

### 端到端對帳結果(2026-06,已驗證)
| 來源 | 總支出 | 筆數 |
|---|---|---|
| 原始記帳 App | 8,239,154 | 127 |
| Supabase(清洗後) | 8,239,154 | 127 |
| 網站(讀 DB 畫圖) | 8,239,154 | 127 |

三邊完全一致 → 清洗無漏、無重複、無分類錯誤,資料可信。

- 6 月收入 2 筆 = 161,600(解約金 160,000 + 退稅 1,600)
- J零用金 2 筆 = 結清 −48,143、撥款 +150,000(不計入費用)
- 6 月依類別:房租金 5,680,681 / 其他 1,978,320 / 裝修清潔 344,996 / 家俱家電 138,140 / 水電網瓦斯 49,230 / 勞健保 27,480 / 生活備品 19,797 / 維修 510

### 「從六月開始實施 UUID」的銜接
Supabase 原有 6 月匯入資料(舊 UUID)會與 Sheet 新發的 UUID 衝突 → 需執行一次 `resetJune`(清掉 DB 的 2026-06)再 `syncRecent` 重灌,讓 Sheet 新 UUID 成為唯一真相。**只做一次。**

## 7. 其他事項

### Supabase 免費方案注意
- **七天無 API 請求會自動暫停**:每晚的 `syncRecent` 會自然「保活」,不會睡著。
- 額度:500MB 儲存、5 萬 MAU、無限 API 請求 → 對本專案綽綽有餘。
- 地區建議選 **Singapore**(離台灣最近)。

### 金鑰管理(務必)
| 金鑰 | 放哪 | 權限 |
|---|---|---|
| **publishable key** (`sb_publishable_…`) | 前端 HTML(可公開) | 唯讀 |
| **secret key** (`sb_secret_…`) | 只放 Apps Script | 讀寫 |
| 資料庫密碼 | 密碼管理器,絕不外流 | 管理 |

> ⚠️ secret key 與資料庫密碼**絕不可**放進前端或 commit 到 GitHub。

### 部署
- 前端是單一 `index.html`,可拖檔上傳或用 `vercel --prod` 部署。
- 檔名必須是 `index.html`(英文),不可中文檔名。
